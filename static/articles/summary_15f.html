Getting Start from Documents

The starting point for me is the wiki pages of Peloton. Before joining the group, all I know about database is SQL, the architecture of database, and some details of how the PostgreSQL implements the join operations.

After reading through the documents, I learned many basic concepts in the field of Database, such as different types of OLTP, OLAP, different storage architectures to deal with these two different types of transections, and what idea Peloton uses to improve performance on both types queries, which were really helpful in the interviews. In addition to that, I also read paper about the column-based database system Vertica The Vertica Analytic Database: C-Store 7 Years Later.

Benchmarking

Then I started to do some benchmarking job, which is easy to start with. I wrote a script to automatically record the benchmarking scores, CPU usage in different circumstance, and visualize the performance.

According to the call graph, we found out the bottle-neck to help make decision on the optimization priority of different modules. In my benchmarking job, I found bugs when getting abnormal scores. I learned to use GDB to debug, however the whole project was too large for me and I wasn't able to figure it out. In the meantime, I had better understanding on how the system was organized, how we reused PostgreSQL to avoid reinventing the wheels.

High Concurrent B+ Tree

The next task is to test a version of High Concurrent B+ Tree and try to integrate it into the Database System. Joy had made some effort on it, and we both thought it should be very easy.

However, the first problem I encountered is that the B+ Tree is written in C, which cannot be compiled with C++ compilers, so I migrated into C++ grammar and did some tests with multi-threads to exclude the incorrect versions.

When I integrated the data structure into the system, I found that the existing interfaces were incompatible with our need. So I dug into to source-code to make modifications. The 2000-line source-code lacked documents and was written by an C expert using many fancy techniques, which was impressive but confusing without fully understanding the design. I spent tens of hours on it but didn't managed to carry out a working version.

Although it was not a successful attempt, I learned many useful techniques and designs from reading and debugging the source-code. One of which is the array of size zero, which could simplify the process of memory management. Another is that in the B+ Tree, for the consideration on efficiency and concurrency, it's unnecessary to delete a leaf-node directly, instead, we can mark the valid bit as zero, which can reduce the operations of node split and merge.

STX B+ Tree

Failing to modify the complicated concurrent B+ Tree, I tried to add simple latching to a B+ Tree structure, STX B+ Tree. The source code is extremely clear, which can be written into textbook.

The simple latching was really simple and I finished the implementation within about two days, but when I took the interfaces we need in our system into account, many question raises. The data structure is not designed for parallel environments, let alone our secondary index application.

Without consideration on concurrency, the access interface are all based on integrators, so it's complicated to manage the locks when we are doing range scan or delete. After reading several papers like A survey of B-tree locking techniques and Efficient Locking for Concurrent Operations on B-Trees and group discussion, I realized that this is not an trivial, on the contrary, it is still relatively open topic. No wonder there's few open-source implementations on the web. Hacking this STX B+Tree in a short time turned out to be beyond my ability. But I still learned a lot about the latching strategies, the design of B+ Tree, and the language techniques in C++, for example, the implementation of iterators.
